# Code Standards

本项目的代码规范保持少而硬：优先保证结构清晰、接口稳定、闭环可测。

## 结构

- 不把业务逻辑堆在根目录单文件里。
- 新能力先归类到明确层级：`core`、`providers`、`tools`、`permissions`、`prompts`、`runtime`、`commands`。
- CLI 只做参数解析和分发，不承载 agent loop 逻辑。
- 设计优先参考 Claude Code 的分层方式，实现优先参考 MiniCode 的轻量方式。

## 类型与接口

- 跨模块数据结构必须用 `pydantic` 或 `dataclass` 定义。
- Provider、Tool、Agent Loop 的边界接口要稳定，不直接传散乱 dict。
- 外部 API 原始响应只允许停留在 adapter 层。

## 异步与错误

- 模型调用、工具执行、子进程、网络请求统一 async。
- 核心路径避免阻塞 I/O；必要时封装到明确模块。
- 默认 fail closed：工具未知、schema 错误、权限不明、配置缺失都应失败并给出清晰错误。
- 不静默吞异常，不让模型绕过权限层执行写文件或命令。

## 测试

- 单元测试覆盖纯逻辑、消息转换、工具注册、错误解析。
- 真实模型行为放在 `tests/integration/`，通过环境变量显式启用。
- 不实现 mock provider、mock 模型或 `--mock` 启动模式；测试替身只能留在测试文件内。
- 文件类测试使用 `tmp_path`。
- 测试优先验证闭环，不追求空洞覆盖率数字。

## 风格

- 使用 `ruff` 统一 lint，行宽 100。
- 函数过长时拆分；超过 80-100 行需重新审视职责。
- 命名使用清晰业务词，少用缩写。
- 注释解释“为什么”，避免复述代码“做了什么”。

## 文档同步

- 新增核心模块时更新对应文档索引或规划说明。
- 新增命令、工具、配置项时补使用说明。
- 实现偏离计划时，先更新计划，再改代码。
